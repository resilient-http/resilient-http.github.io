Resilient

Specification Clients Github

A fault tolerant, dynamic server discovery, balanced and full featured HTTP client for distributed and reactive systems

The client of a system should not be burdened with handling its failures. It should be reliable, responsive, elastic and fault tolerant. Resilient is the smart HTTP client for the new era of reactive systems architectures.

Learn how it works or see the clients and compatible servers

Introduction

Organisations working in disparate domains are independently discovering patterns for building software that look the same. These systems are more robust, more resilient, more flexible and better positioned to meet modern demands.

The system stays responsive in the face of failure. This applies not only to highly-available, mission critical systems — any system that is not resilient will be unresponsive after a failure

What resilient specifically means?

Acording to the reactive manifesto, resilience is achieved by replication, containment, isolation and delegation. Failures are contained within each component, isolating components from each other and thereby ensuring that parts of the system can fail and recover without compromising the system as a whole.

Recovery of each component is delegated to another (external) component and high-availability is ensured by replication where necessary. The client of a service is not burdened with handling its failures

Be much responsible as possible

Resilient controls the request control flow in a transversal way in order to guarantee the most successful consistency in your HTTP traffic, with exceptional handling for errors and fallbacking to the next best available server

Server discovering and caching

Resilient has built-in support for dynamic servers discovering, refreshing and caching. You could provide a list of servers which gives to Resilient a list of available service servers. Discovery servers must implement the Resilient interface specification

Client side balancing

Resilient aims to delegate the balancing logic responsabilities in a proper and consistent way on the client-side, providing an extra layer of reliability from the consumer side, taking into account network latency, networks errors, server errors, load distribution... in order to use the best available server

Polyglot

Resilient wants to become a multi-language HTTP client. There are plans to port the JavaScript client implementation to multiple languages such as Python, Ruby, Go, Clojure... in a near future, after the client specification have more madurity
